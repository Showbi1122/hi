  - question: "What is the difference between SSG and SSR in Next.js?"
    answer: "SSG (Static Site Generation) builds HTML at deploy time and serves it from a CDN. SSR (Server-Side Rendering) generates HTML on each request when a user visits. SSG is faster and cheaper at scale for content that does not change every second; SSR fits personalized or highly dynamic pages."
  - question: "What is ISR in Next.js?"
    answer: "Incremental Static Regeneration (ISR) keeps static pages fast but refreshes them on a schedule or on demand. You get CDN speed with content that updates without rebuilding the entire site. Common for product catalogs, news, and large e-commerce inventories."
  - question: "When should I use Client-Side Rendering (CSR)?"
    answer: "Use CSR for authenticated, highly interactive UI where SEO does not matter — dashboards, settings panels, notifications, chat widgets. In the App Router, mark components with \"use client\" and keep the page shell server-rendered when possible."
  - question: "Does SSR improve SEO compared to CSR?"
    answer: "For public pages that must rank on Google, yes. SSR and SSG deliver complete HTML to crawlers immediately. CSR-only pages rely on JavaScript execution, which is slower and easier to get wrong for indexing."
  - question: "Can I mix SSG, SSR, ISR, and CSR in one Next.js app?"
    answer: "Yes, and you should. The App Router encourages per-route decisions. Marketing pages can be static, product pages ISR, user dashboards CSR inside a server layout, and admin reports SSR."
  - question: "How does Next.js 15 App Router handle static pages?"
    answer: "Routes are static by default when they use only Server Components and do not call dynamic APIs like cookies(), headers(), or uncached fetch. Export const dynamic = 'force-static' or generateStaticParams makes intent explicit for dynamic segments."
  - question: "Is ISR still relevant in Next.js 15?"
    answer: "Yes. Use revalidate in fetch options or export const revalidate on a page/route segment. ISR remains the sweet spot for large catalogs where full SSG rebuilds are too slow but SSR on every hit is too expensive."
  - question: "What hurts Core Web Vitals in Next.js?"
    answer: "Oversized Client Components, unoptimized images, blocking third-party scripts, SSR without caching on high-traffic pages, and shipping the same huge JavaScript bundle to every route. Pick SSG or ISR for LCP-sensitive marketing pages."
  - question: "Should a POS system use SSR or CSR?"
    answer: "Often both. Public marketing pages: SSG. Authenticated POS terminal UI: CSR or client-heavy islands for speed of interaction. Reports with sensitive per-user data: SSR or SSR with short cache disabled."
  - question: "How do I choose between SSG and ISR for a blog?"
    answer: "Small blogs with infrequent posts: SSG at build time is simplest. Large blogs or multi-author sites with constant publishing: ISR or on-demand revalidation avoids redeploying for every article."

rendering" article that ends with "it depends," you are not alone. The frustrating truth is that it <em>does</em> depend — but not in a vague way. It depends on whether strangers need to find the page on Google, how often data changes, whether the user is logged in, and how much you are willing to pay in server compute.</p> <p>Next.js is not one rendering mode. It is a router that lets you compose <strong>Server Components</strong>, <strong>Client Components</strong>, static HTML, dynamic HTML, and time-based revalidation in the same application. The App Router (Next.js 13+) made Server Components the default, which confused teams who learned Next.js through the Pages Router <code>getServerSideProps</code> / <code>getStaticProps</code> split. The mental model shifted: instead of picking a data function per page, you design component trees and caching boundaries.</p> <p>This guide is for developers shipping real products — portfolios, <a href="/services#service-industry">POS systems</a>, school management software, CRMs, marketing sites, and SaaS dashboards. I will compare <strong>CSR</strong>, <strong>SSG</strong>, <strong>SSR</strong>, and <strong>ISR</strong> the way I explain them on client calls: with examples, trade-offs, and a decision flow you can paste into your architecture doc.</p> </section> <section class="article-section reveal" id="why-rendering-matters"> <h2>Why rendering strategy matters</h2> <p>Rendering is where performance, SEO, infrastructure cost, and developer experience collide. Pick the wrong default and you will fight the framework for months.</p> <h3>What goes wrong when you guess</h3> <ul> <li><strong>All CSR:</strong> Marketing pages do not rank, LCP suffers, social previews break.</li> <li><strong>All SSR:</strong> Server bills spike, TTFB grows under load, you cache nothing and wonder why Vercel costs jumped.</li> <li><strong>All SSG:</strong> Stale product inventory, personalized dashboards impossible without hacks.</li> <li><strong>ISR misconfigured:</strong> Users see outdated prices for an hour because revalidate was set to 3600 without on-demand hooks.</li> </ul> <p>Rendering strategy is an architecture decision, not a tutorial checkbox. The teams that win treat each route as its own micro-product with its own freshness and visibility requirements.</p> <blockquote class="expert-insight"> <p>I audited a school management startup that SSR'd every route — including the public homepage — because one developer copied a tutorial. Their homepage TTFB was 800ms on mobile. We moved marketing to SSG and kept the admin timetable view on SSR. Same codebase, no feature loss, hosting cost dropped roughly 40%.</p> </blockquote> </section> <section class="article-section reveal" id="understanding-csr"> <h2>Understanding CSR (Client-Side Rendering)</h2> <p><strong>Client-Side Rendering</strong> means the browser downloads JavaScript, executes it, and builds the DOM. In React SPAs (Vite + React Router), essentially everything is CSR. In Next.js App Router, any component file with <code>"use client"</code> hydrates on the client — that is CSR for those subtrees.</p> <h3>How CSR works in Next.js 15</h3> <p>Server Components render on the server and send HTML. Client Components send HTML placeholders plus JavaScript bundles that hydrate interactivity. If you mark an entire page as a Client Component, you have turned that page into a mini-SPA for hydration purposes, even if the route itself was server-rendered once.</p> <h3>Real projects where CSR shines</h3> <ul> <li><strong>Dashboard analytics:</strong> Charts, filters, date ranges — heavy interaction, no SEO.</li> <li><strong>Settings pages:</strong> Profile forms, notification toggles, API keys.</li> <li><strong>Notifications center:</strong> Real-time lists, mark-as-read, WebSocket updates.</li> <li><strong>Chat systems:</strong> Message threads, typing indicators, presence.</li> <li><strong>POS terminal UI:</strong> After staff login, barcode scanning and cart updates must feel instant; SEO is irrelevant.</li> </ul> <h3>CSR trade-offs</h3> <p>CSR feels fast <em>after</em> load. The first visit still pays the JavaScript tax. For public pages, that tax hits SEO and Core Web Vitals. Use CSR deliberately as islands, not as the outer shell of crawlable content.</p> </section> <section class="article-section reveal" id="understanding-ssg"> <h2>Understanding SSG (Static Site Generation)</h2> <p><strong>Static Site Generation</strong> pre-renders HTML at build time. Users receive files from a CDN edge — no Node server required per request. In the App Router, routes are static when they do not use dynamic server APIs and use cached data fetching.</p> <h3>How to make a route static in App Router</h3> <ul> <li>Use Server Components without <code>cookies()</code>, <code>headers()</code>, or <code>searchParams</code> forcing dynamic.</li> <li>Use <code>export const dynamic = 'force-static'</code> when you want to be explicit.</li> <li>For dynamic segments, implement <code>generateStaticParams()</code> — this portfolio's <code>/blog/[slug]</code> and <code>/geo/[slug]</code> routes do exactly that.</li> <li>Run <code>next build</code> and confirm the output table shows <code>○ (Static)</code> for your routes.</li> </ul> <h3>Real projects where SSG shines</h3> <ul> <li><strong>Portfolio websites:</strong> Like this site — 28 prerendered routes, fast globally.</li> <li><strong>Company websites:</strong> About, services, contact — copy changes weekly, not every second.</li> <li><strong>Documentation:</strong> Versioned docs, MDX content, minimal server logic.</li> <li><strong>Marketing websites:</strong> Landing pages, pricing, feature grids.</li> <li><strong>Blogs:</strong> Articles published occasionally; rebuild or ISR on publish.</li> </ul> <p>SSG is the highest ROI mode for <a href="/blog/seo-best-practices-business-websites">Next.js SEO</a> on content sites: crawlers get full HTML, TTFB is tiny, and hosting is cheap.</p> </section> <section class="article-section reveal" id="understanding-ssr"> <h2>Understanding SSR (Server-Side Rendering)</h2> <p><strong>Server-Side Rendering</strong> generates HTML on each request (or serves from a short-lived cache). Use it when the response must reflect request-specific or frequently changing data that cannot be predicted at build time.</p> <h3>What forces dynamic rendering in App Router</h3> <ul> <li><code>export const dynamic = 'force-dynamic'</code></li> <li>Reading cookies or headers for personalization</li> <li><code>fetch(url, { cache: 'no-store' })</code> or uncached database queries</li> <li>Search results pages driven by query strings that must be server-rendered for SEO</li> </ul> <h3>Real projects where SSR shines</h3> <ul> <li><strong>Admin dashboards:</strong> Role-based sidebars, per-tenant configuration.</li> <li><strong>POS back-office:</strong> Shift reports, staff permissions, store-specific pricing.</li> <li><strong>School management:</strong> Timetables and attendance filtered by term and campus.</li> <li><strong>CRM:</strong> Pipeline views with live deal stages from the database.</li> <li><strong>ERP modules:</strong> Inventory snapshots that must be current when the page loads.</li> <li><strong>User profile (semi-public):</strong> When content varies per session.</li> </ul> <p>SSR is not "more professional" than SSG — it is more <em>dynamic</em>. Dynamic costs money and latency. Pay for it only where the product requires it.</p> </section> <section class="article-section reveal" id="understanding-isr"> <h2>Understanding ISR (Incremental Static Regeneration)</h2> <p><strong>Incremental Static Regeneration</strong> combines static delivery with background updates. A page is static until its revalidation window expires; then the next request triggers a rebuild of that page.</p> <h3>ISR in Next.js 15 App Router</h3> <p>Set revalidation at the route or fetch level:</p> <ul> <li><code>export const revalidate = 60</code> — regenerate at most once per 60 seconds per path.</li> <li><code>fetch('https://api.example.com/products', { next: { revalidate: 300 } })</code></li> <li>On-demand revalidation via <code>revalidatePath()</code> or <code>revalidateTag()</code> after CMS publish webhooks.</li> </ul> <h3>Real projects where ISR shines</h3> <ul> <li><strong>News websites:</strong> Homepage refreshes frequently without full redeploys.</li> <li><strong>Product catalogs:</strong> Thousands of SKUs; rebuilding all pages on every stock change is wasteful.</li> <li><strong>E-commerce:</strong> Product detail pages with hourly price or availability updates.</li> <li><strong>Service listings:</strong> Directories, job boards, rental listings updated throughout the day.</li> </ul> <p>ISR is the pattern many teams need but do not know the name for. They SSR product pages "to be safe" and pay 10x server cost compared to ISR with on-demand revalidation when inventory changes.</p> </section> <section class="article-section reveal" id="performance-comparison"> <h2>Performance comparison</h2> <p>Raw speed rankings for <em>typical</em> implementations (your mileage varies with discipline):</p> <ol> <li><strong>SSG</strong> — HTML from CDN edge, minimal compute, best TTFB globally.</li> <li><strong>ISR (cache hit)</strong> — Same as SSG until revalidation; occasional slower request during regeneration.</li> <li><strong>SSR (cached)</strong> — Good if you use CDN or data cache; poor if every request hits the database cold.</li> <li><strong>CSR (first load)</strong> — Waits on JS parse/execute; can be OK after code-splitting but rarely beats static HTML for LCP.</li> </ol> <h3>After first load</h3> <p>CSR and hydrated Client Components feel snappy for in-app navigation. That is why dashboards tolerate CSR. Public sites live or die on first load from Google Ads or organic search — optimize for cold visits.</p> <h3>Measuring Next.js performance</h3> <p>Run Lighthouse on throttled mobile, check WebPageTest filmstrips, and read Vercel Analytics or your APM. Compare the same page under SSG vs SSR with real data — not blog benchmarks. I ship <code>app/sitemap.ts</code> and static routes first, then add dynamic modes only where metrics prove the need.</p> </section> <section class="article-section reveal" id="seo-comparison"> <h2>SEO comparison</h2> <p>For <strong>Next.js SEO</strong>, crawlers want complete HTML, stable URLs, unique metadata, and fast loads. Ranking also needs content quality — rendering only removes technical blockers.</p> <h3>SEO by rendering mode</h3> <ul> <li><strong>SSG / ISR:</strong> Excellent. HTML is immediate; Metadata API titles and descriptions are in the document. Best for public content.</li> <li><strong>SSR:</strong> Excellent for indexable dynamic routes (e.g. public profile URLs, filtered listing pages you want indexed).</li> <li><strong>CSR-only public pages:</strong> Poor default. Google can render JS, but it is slower and brittle for smaller teams.</li> </ul> <p>Use the <strong>Metadata API</strong> in <code>layout.tsx</code> or <code>page.tsx</code> for titles, descriptions, Open Graph, and Twitter cards. Pair with JSON-LD, <code>app/sitemap.ts</code>, and <code>app/robots.ts</code> — not a hand-rolled static <code>sitemap.xml</code> you forget to update.</p> <p>For deeper checklists, see <a href="/blog/seo-best-practices-business-websites">SEO best practices for business websites</a> and <a href="/blog/react-vs-nextjs">React vs Next.js</a> for when CSR SPAs fight indexing.</p> </section> <section class="article-section reveal" id="server-cost"> <h2>Server cost comparison</h2> <p>Infrastructure cost scales with <em>dynamic compute per request</em>.</p> <ul> <li><strong>SSG:</strong> Lowest. CDN bandwidth only on many hosts.</li> <li><strong>ISR:</strong> Low. Occasional regeneration work; still mostly CDN.</li> <li><strong>SSR:</strong> Highest at scale. Every uncached request needs a serverless function or Node process.</li> <li><strong>CSR:</strong> Hosting can be cheap (static assets), but users pay in load time; API backends still cost money.</li> </ul> <p>A viral marketing page on SSR can spike bills. The same page on SSG absorbs traffic like a CDN sponge. A POS admin SSR report used by twelve store managers is fine; a public homepage SSR'd for a million visitors is not.</p> </section> <section class="article-section reveal" id="caching"> <h2>Caching differences</h2> <p>Next.js 15 caching is explicit and sometimes surprising. Understand the layers:</p> <dl class="definition-box"> <dt>Full Route Cache</dt> <dd>Static routes stored at build time. SSG and ISR static output live here.</dd> <dt>Data Cache</dt> <dd><code>fetch</code> responses cached by default unless opted out. Control with <code>cache</code>, <code>next.revalidate</code>, and tags.</dd> <dt>Request Memoization</dt> <dd>Deduplicates identical fetch calls within one request — not across users.</dd> <dt>Router Cache (client)</dt> <dd>Client-side RSC payload cache during navigation — affects UX, not SEO crawlers.</dd> </dl> <h3>Practical caching rules</h3> <ul> <li>Marketing content: cache aggressively (SSG or long ISR).</li> <li>User-specific dashboards: <code>cache: 'no-store'</code> or dynamic rendering.</li> <li>Catalog pages: ISR + on-demand revalidation when products change.</li> <li>Never cache personalized HTML at the CDN without varying on cookies — you will leak data.</li> </ul> </section> <section class="article-section reveal" id="core-web-vitals"> <h2>Core Web Vitals impact</h2> <p>Google's Core Web Vitals tie ranking and UX to measurable thresholds.</p> <h3>LCP (Largest Contentful Paint)</h3> <p>SSG/ISR usually win — hero images and headings are in the first HTML byte stream. CSR delays LCP until JS runs. Use <code>next/image</code> with <code>priority</code> on above-the-fold images only.</p> <h3>INP (Interaction to Next Paint)</h3> <p>CSR-heavy pages can score well on INP <em>after</em> hydration if bundles are small. Bloated Client Components hurt INP everywhere. Split interactive widgets; do not hydrate the entire layout.</p> <h3>CLS (Cumulative Layout Shift)</h3> <p>Rendering mode matters less than reserving image dimensions and font metrics. Server-rendered HTML with explicit width/height on media still wins for perceived stability.</p> <p>Rendering strategy is one lever. Image optimization, font loading, and third-party script discipline are the others. I treat all three in <a href="/services#service-seo">SEO-optimized builds</a>.</p> </section> <section class="article-section reveal" id="which-strategy"> <h2>Which strategy should you choose?</h2> <p>Stop choosing globally. Map user journeys:</p> <ol> <li>Does Google need to index this URL?</li> <li>Does the HTML change per user session?</li> <li>How often does data change?</li> <li>How interactive is the UI?</li> </ol> <p>If (1) is yes and (2) is no → lean SSG or ISR. If (2) is yes → SSR or CSR inside auth. If (4) is extreme → CSR islands. If (3) is "every few minutes" → ISR. If (3) is "real-time" → SSR or client fetch with loading states.</p> </section> <section class="article-section reveal" id="comparison-table"> <h2>Full comparison table</h2> <table class="compare-table"> <thead> <tr> <th>Factor</th> <th>CSR</th> <th>SSG</th> <th>SSR</th> <th>ISR</th> </tr> </thead> <tbody> <tr> <td>SEO</td> <td>Poor for public pages</td> <td>Excellent</td> <td>Excellent</td> <td>Excellent</td> </tr> <tr> <td>Performance (TTFB)</td> <td>Slower first load</td> <td>Fastest</td> <td>Moderate</td> <td>Fast (stale until revalidate)</td> </tr> <tr> <td>Server load</td> <td>Low (static assets)</td> <td>Very low</td> <td>High</td> <td>Low–medium</td> </tr> <tr> <td>Build time</td> <td>Low</td> <td>Grows with page count</td> <td>Low</td> <td>Medium (partial builds)</td> </tr> <tr> <td>Caching</td> <td>Browser + API caches</td> <td>CDN indefinite</td> <td>Must configure; often none</td> <td>CDN + time/tag revalidation</td> </tr> <tr> <td>Dynamic data</td> <td>Client fetches after load</td> <td>Build-time only</td> <td>Per request</td> <td>Stale-while-revalidate</td> </tr> <tr> <td>Best use cases</td> <td>Dashboards, chat, settings</td> <td>Portfolio, docs, marketing</td> <td>Admin, CRM, personalized views</td> <td>News, e-commerce, catalogs</td> </tr> <tr> <td>Advantages</td> <td>Rich interactivity</td> <td>Speed, cost, SEO</td> <td>Fresh per-request data</td> <td>Speed + fresher content</td> </tr> <tr> <td>Disadvantages</td> <td>SEO, first paint</td> <td>Stale until rebuild</td> <td>Cost, latency</td> <td>Complexity, brief staleness</td> </tr> </tbody> </table> </section> <section class="article-section reveal" id="decision-flow"> <h2>Decision flow: CSR vs SSG vs SSR vs ISR</h2> <p>Walk this flow top to bottom for each route:</p> <ol> <li><strong>Must this URL rank in search?</strong> No → CSR is fine behind login (dashboard, POS terminal, analytics). Yes → continue.</li> <li><strong>Is content identical for all visitors?</strong> Yes → continue. No → SSR (or static shell + CSR island for private widgets).</li> <li><strong>Does data change between deploys?</strong> No → <strong>SSG</strong> (portfolio, company site, blog with rebuild on publish).</li> <li><strong>Does data change frequently but tolerate short staleness?</strong> Yes → <strong>ISR</strong> (products, listings, news).</li> <li><strong>Must every request see live data?</strong> Yes → <strong>SSR</strong> with careful caching (admin reports, per-user CRM).</li> <li><strong>Is the page mostly interactive UI?</strong> Use Server Components for the shell; add <strong>CSR</strong> Client Components for charts, maps, and forms.</li> </ol> <p>That is the entire game. Everything else is implementation detail.</p> </section> <section class="article-section reveal" id="common-mistakes"> <h2>Common mistakes developers make</h2> <p><strong>Using SSR everywhere.</strong> Tutorial cargo-culting. Marketing pages do not need per-request HTML.</p> <p><strong>Using CSR for SEO pages.</strong> Landing pages built as pure Client Components with empty initial HTML. Fix: Server Component page + small client islands.</p> <p><strong>Ignoring caching.</strong> Uncached database queries on SSR pages under load. Fix: <code>revalidate</code>, React <code>cache()</code>, or move to ISR.</p> <p><strong>Poor metadata.</strong> One generic <code>&lt;title&gt;</code> for all routes. Fix: Metadata API per page — see <code>lib/metadata.ts</code> patterns.</p> <p><strong>Missing sitemap.</strong> Crawlers discover pages slowly. Fix: dynamic <code>app/sitemap.ts</code> from your content source.</p> <p><strong>Missing robots.txt.</strong> Fix: <code>app/robots.ts</code> with sitemap reference.</p> <p><strong>Not optimizing images.</strong> Hero JPEGs at 4000px width kill LCP. Fix: <code>next/image</code>, AVIF/WebP, <code>sizes</code> attribute.</p> <p><strong>Ignoring Core Web Vitals.</strong> Shipping 400KB Client Components to every route. Fix: analyze bundle, dynamic import heavy charts.</p> <p><strong>"use client" at the root.</strong> Wrapping entire layouts in Client Components throws away RSC benefits.</p> <p><strong>Choosing ISR without a revalidation plan.</strong> Stale prices during sales events. Fix: on-demand <code>revalidatePath</code> from webhooks.</p> </section> <section class="article-section reveal" id="best-practices"> <h2>Best practices for Next.js 15 App Router</h2> <h3>Architecture</h3> <ul> <li>Default to Server Components; push <code>"use client"</code> to leaves of the tree.</li> <li>Colocate data fetching in Server Components; pass serializable props to clients.</li> <li>Use <code>generateStaticParams</code> for all known dynamic slugs (blog, products, docs).</li> <li>Document rendering mode per route in README or ADR — future you will thank you.</li> </ul> <h3>Performance &amp; SEO</h3> <ul> <li>Static marketing, dynamic product — split routes, not repos.</li> <li>Lazy-load modals and charts with <code>next/dynamic</code> and <code>ssr: false</code> only when necessary.</li> <li>Export metadata from every public <code>page.tsx</code>; include Open Graph and canonical URLs.</li> <li>Generate sitemap and robots from code, not manual XML files.</li> <li>Measure CWV before launch; set performance budgets in CI if the team is mature enough.</li> </ul> <h3>Operations</h3> <ul> <li>Wire CMS/webhook → <code>revalidatePath</code> for ISR catalogs.</li> <li>Keep secrets and DB access in Server Components or Route Handlers only.</li> <li>Use environment-based <code>NEXT_PUBLIC_SITE_URL</code> for absolute URLs in metadata.</li> </ul> <p>These are the patterns I use when shipping <a href="/services#service-saas">SaaS MVPs</a> and business software — the <a href="/blog/saas-development-guide">SaaS development guide</a> covers scope;
